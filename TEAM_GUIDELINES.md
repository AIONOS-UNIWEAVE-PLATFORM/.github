# UniWeave Engineering Guidelines\n\n**For**: Nomaan, Upender, Himanshu, and all future UniWeave engineers\n**From**: Agam (Head of Product)\n**Date**: March 5, 2026\n\n---\n\n## 1. What changed\n\nWe've created a new GitHub org: **AIONOS-UNIWEAVE-PLATFORM**\n\nThis is the permanent home for all UniWeave engineering work. Previously, our code lived across two orgs (UNIWEAVE-AIONOS and Voice-AI-Project) with no shared standards. The new org gives us:\n\n- One place for all repos\n- Consistent branch protection and CI\n- Shared PR templates and conventions\n- AI-native development standards (CLAUDE.md in every repo)\n\n**Structure**: No teams — everyone is an org member with `write` access. Agam is org owner. Simple and flat. We're 4-5 people, not 50.\n\n**What happens to old repos**: They stay where they are during transfer. Once a repo is transferred to the new org, the old location auto-redirects. Nothing breaks. Old orgs become archives — no new repos created there.\n\n---\n\n## 2. How we work\n\n### Branch workflow\n\n```\nmain (protected) <- PR required\n  |-- feat/short-description\n  |-- fix/short-description\n  |-- chore/short-description\n  |-- docs/short-description\n```\n\n- **Never push directly to `main`**. Always open a PR.\n- Branch from `main`, PR back to `main`.\n- Keep branches short-lived — merge within 2-3 days.\n- Delete branches after merge.\n\n### Commit conventions\n\nFormat: `<type>(<scope>): <description>`\n\nTypes: `feat`, `fix`, `chore`, `docs`, `refactor`, `test`, `ci`\n\nExamples:\n```\nfeat(chat-engine): add Azure OpenAI streaming endpoint\nfix(whatsapp): handle empty message body gracefully\nchore(deps): bump express to 4.19\ndocs(readme): add local setup instructions\n```\n\n**AI-assisted work**: Add `[ai-assisted]` at the end of the commit message when Claude Code or any AI tool helped write the code:\n\n```\nfeat(pulse): add post-call webhook handler [ai-assisted]\n```\n\nThis is not a mark of shame — it's a signal for reviewers to focus on intent and tests rather than line-by-line style.\n\n### PR process\n\nEvery PR uses the org template (auto-loaded). Fill in:\n\n1. **What changed** — 1-2 sentences\n2. **Why** — the business reason or ticket reference\n3. **How to test** — steps a reviewer can follow\n4. **AI-assisted checkbox** — check if AI helped write code\n5. **Trello card link** — connect to the board\n\nPRs need **1 approval** before merge. Agam can approve.\n\n### Code review philosophy\n\n- **Review intent and tests**, not style. Linters handle formatting.\n- For AI-assisted code: verify (a) it does what the description says, (b) tests are meaningful, (c) no security issues.\n- Leave comments as suggestions, not demands. Use GitHub's \"suggestion\" feature.\n\n---\n\n## 3. CLAUDE.md — your repo's instruction manual\n\nEvery repo gets a `CLAUDE.md` file at the root. This is the first thing Claude Code reads when it opens your project. Think of it as onboarding docs that work for both humans AND AI agents.\n\n### Why it matters\n\n- Claude Code reads it automatically — no setup required\n- New team members (human or AI) can be productive in minutes\n- It's the single source of truth for \"how does this service work\"\n- It prevents Claude from guessing wrong about your codebase\n\n### What goes in yours\n\nSee the [CLAUDE.md template](CLAUDE_MD_TEMPLATE.md). Key sections:\n- What the service does (3 sentences max)\n- How to run locally\n- How to test\n- Key files (top 5 files and what they do)\n- Environment variables\n- Architecture notes\n- Dependencies on other services\n\n### Rules\n\n- Keep it accurate. If you change how the service runs, update CLAUDE.md.\n- PR changes to CLAUDE.md like code — it's a living document.\n- Don't put secrets in it.\n- Maximum ~100 lines. Be concise.\n\n---\n\n## 4. Your first task — repo onboarding\n\nEach engineer transfers their assigned repo(s) and opens an onboarding PR.\n\n### Step 1: Transfer your repo\n\nGo to your repo > Settings > General > Danger Zone > Transfer repository > Transfer to `AIONOS-UNIWEAVE-PLATFORM`.\n\nYou need admin rights on the repo to transfer it. If you don't have them, ask Agam.\n\n### Step 2: Write your CLAUDE.md\n\nUse the [template](CLAUDE_MD_TEMPLATE.md). Fill in every section. If you're not sure about something, put \"TBD\" — don't leave it blank.\n\n### Step 3: Ensure README.md exists\n\nEvery repo needs a README with:\n- Service name and one-line description\n- Prerequisites (Node version, Python version, etc.)\n- Setup instructions (`npm install`, `pip install -r requirements.txt`, etc.)\n- How to run locally\n- How to run tests\n- Environment variables needed (names only — never commit values)\n\n### Step 4: Add .env.example\n\nList every environment variable the service needs, with placeholder values:\n```\nAZURE_OPENAI_ENDPOINT=https://your-endpoint.openai.azure.com/\nAZURE_OPENAI_KEY=your-key-here\nMONGODB_URI=mongodb://localhost:27017/your-db\nKAFKA_BOOTSTRAP_SERVERS=localhost:9092\n```\n\n**Never commit `.env` files.** Only `.env.example`.\n\n### Step 5: Verify .gitignore\n\nMake sure `.gitignore` includes:\n```\nnode_modules/\n.env\n.env.local\n*.log\ndist/\nbuild/\n__pycache__/\n.pytest_cache/\n```\n\n### Step 6: Open your onboarding PR\n\nBranch: `chore/onboarding`\nPR title: `chore: add CLAUDE.md, README, .env.example for repo onboarding`\n\nThis is your first PR in the new org. It sets the foundation for everything else.\n\n### Repo transfer assignments\n\n| Engineer | Repo(s) to transfer | Priority |\n|---|---|---|\n| **Nomaan** | `intelliconverse-livekit-frontend` | After #94 Chat Engine milestone |\n| **Upender** | `analytics-user-mgmt` | After #96 IntelliPulse milestone |\n| **Himanshu** | `intelli-converse-tools-service` | After #98 IntelliRAG milestone |\n| **Agam** | `voice-prompt-builder`, `intelliconverse-evals` | This week |\n| **New** | `ai-gateway` (Agam creates fresh in new org) | When needed |\n\n**Note**: Transfer after your current sprint card milestone — don't interrupt in-progress work.\n\n---\n\n## 5. Repo standards\n\n### Directory conventions\n\nNo enforced monorepo structure. Each service owns its layout. But follow these basics:\n\n- `src/` or `app/` — application code\n- `test/` or `__tests__/` — test files\n- `.github/workflows/` — CI pipelines (use org templates as starting point)\n- `CLAUDE.md` — at the root, always\n- `README.md` — at the root, always\n- `.env.example` — at the root, always\n- `.gitignore` — at the root, always\n\n### CI templates\n\nThe org provides starter CI workflows:\n- **Node.js**: checkout > setup-node > npm ci > lint > test > build\n- **Python**: checkout > setup-python > pip install > ruff check > pytest\n\nCopy the relevant template from [workflow-templates/](workflow-templates/) into your repo's `.github/workflows/` and customize.\n\n---\n\n## 6. AI-native development rules\n\nWe all use Claude Code. These rules make AI-assisted development transparent and safe.\n\n### The `[ai-assisted]` tag\n\nAdd it to any commit where AI wrote or substantially modified the code. This is:\n- **Required** — not optional\n- **Not a negative signal** — it's a review hint\n- **Applied per-commit** — if some commits in a PR are AI-assisted and others aren't, tag only the relevant ones\n\n### Intent-based review\n\nWhen reviewing AI-assisted PRs:\n1. Does the PR description clearly state what changed and why?\n2. Do the tests cover the stated behavior?\n3. Are there obvious security issues (hardcoded secrets, SQL injection, etc.)?\n4. Does the approach make architectural sense?\n\nDon't review AI code line-by-line for style. Review it for correctness and intent.\n\n### Tests as quality gate\n\n- Every new feature needs at minimum: one happy-path test + one error-path test\n- Tests must pass before merge\n- If you can't write a test for something, explain why in the PR description\n\n---\n\n## 7. Security basics\n\n- **2FA required** — enable two-factor authentication on your GitHub account\n- **No secrets in code** — use `.env` files locally, environment variables in deployment. Never commit API keys, tokens, or passwords.\n- **`.env` in `.gitignore`** — always. Use `.env.example` for documentation.\n- **Report if you see secrets** — if you find a committed secret, tell Agam immediately. We'll rotate the credential and clean git history.\n\n---\n\n## 8. What's NOT changing\n\n- **Tech stack stays polyglot** — Java, Node.js, Python all welcome. No forced migrations.\n- **No code refactoring required** — transfer repos as-is. Clean up over time through normal PRs.\n- **Old repos archived, not deleted** — history preserved. Redirects work automatically after transfer.\n- **Trello stays** — no migration to GitHub Projects.\n- **Deployment process stays the same** — Helm charts, same infra. CI is additive, not replacing anything.\n- **Your current sprint work continues** — transfer repos AFTER your current milestone, not during.\n\n---\n\n## Questions?\n\nAsk Agam. This is a one-time setup — once your repo is transferred and onboarding PR is merged, you're done. Everything after that is normal development.
